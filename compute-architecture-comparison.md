# NPU 计算架构风格深度对比

> 目标：为 0.2 TOPS (200 GOPS)、INT8/INT16、CNN-only、MCU级别 NPU 选择最优计算架构
> 首先在 FPGA 上验证，后续可能流片 ASIC

---

## 一、架构概述与工作原理

### 1. 脉动阵列 (Systolic Array)

**代表项目**：Google TPU、UC Berkeley Gemmini、tinyTPU

**工作原理**：
- 由规则排列的 PE（Processing Element）组成 M×N 二维网格
- 数据像"脉搏"一样在相邻 PE 间同步流动，每个 PE 执行一次 MAC 操作后将结果传递给下一个 PE
- 无需全局互连，仅依赖近邻通信，极大简化布线
- 通过 im2col 将卷积转换为矩阵乘法后映射到阵列

**数据流模式**：

| 模式 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **Weight Stationary (WS)** | 权重固定在PE中，输入/输出流过阵列 | 权重复用最大化，推理友好，控制简单 | 对动态网络不灵活 |
| **Output Stationary (OS)** | 输出固定在PE中，权重和输入流过 | 减少部分和写回，适合大batch | 权重重复加载，带宽压力大 |
| **Row Stationary (RS)** | 每行PE维护特定的1D卷积行 | 同时优化所有数据类型的复用，最优能效 | 控制逻辑复杂，映射难度大 |

**适用 0.2 TOPS 的典型规模**：
```
Peak TOPS = M × N × 2 × Freq / 10^12

目标 0.2 TOPS @ 200MHz:
  M × N = 0.2T / (2 × 200M) = 500 PEs
  → 推荐 16×32 或 24×24 的阵列

目标 0.2 TOPS @ 100MHz（FPGA 友好频率）:
  M × N = 0.2T / (2 × 100M) = 1000 PEs
  → 推荐 32×32 的阵列
```

---

### 2. SIMD/向量处理器 (SIMD/Vector Processor)

**代表项目**：ARM Ethos-U55、CEVA NeuPro、Cadence Tensilica

**工作原理**：
- 基于指令驱动的向量计算单元，多个 MAC 单元组成向量通道
- 通过"命令流"（Command Stream）调度计算，类似微控制器但高度专用化
- 主处理器（如 Cortex-M）编译模型生成命令序列，NPU 硬件解码并执行

**Ethos-U55 架构要点**：
- 配置选项：32/64/128/256 个 8×8 MAC/cycle
- 128 MAC @ 500MHz = 128 GOPS（INT8 下约 0.128 TOPS）
- 256 MAC @ 500MHz = 256 GOPS（INT8 下约 0.256 TOPS）
- 面积约 0.1 mm²（16nm），极度面积优化
- 支持权重压缩（减少内存带宽需求）
- 支持固定算子集：Conv2D、DepthwiseConv、FC、Pooling、ReLU、Sigmoid 等
- 通过 2 个 AXI Master 接口访问 SRAM 和 Flash

**TOPS 计算**：
```
Ethos-U55 256MAC @ 500MHz:
  256 × 2 × 500M / 10^12 = 0.256 TOPS

我们目标 0.2 TOPS @ 200MHz:
  需要 500 MAC lanes（向量宽度 500）
  或 250 MAC lanes @ 400MHz
```

---

### 3. 数据流架构 (Dataflow Architecture)

**代表项目**：MIT Eyeriss v1/v2、NVIDIA NVDLA、ShiDianNao

**工作原理**：
- 空间映射 + 显式数据复用优化
- 每个 PE 有本地 Scratchpad 存储，通过 NoC（片上网络）或专用总线连接
- 编译器/映射工具决定数据在空间上的分布，最大化局部数据复用
- 不同于脉动阵列的固定数据流方向，可根据层类型灵活调整

**Eyeriss Row Stationary 详解**：
- 168 个 PE，每个 PE 含本地 SPad (448B)
- 将 1D 卷积（一行 filter × 一行 ifmap → 一行 psum）映射到单个 PE
- 同时复用 filter weight、ifmap pixel 和 partial sum
- 能效：在 65nm 下实现约 0.24 TOPS/W（AlexNet），远优于同期 GPU

**NVDLA 架构**：
- 模块化设计：卷积核 + SDP（激活/BN）+ PDP（池化）+ CDP（LRN）+ Reshape
- Small 配置：64 MAC，约 80K LUT @ FPGA 100MHz
- 可配置 MAC 阵列：C×K（C: 8-64 输入通道并行，K: 4-64 输出通道并行）
- 支持 Fused Mode（流水线融合，减少中间结果回写 DRAM）

**NVDLA 面积参考（28nm ASIC）**：

| MAC数 | Conv Buffer | 面积 (mm²) | INT8 ResNet-50 fps @1GHz |
|-------|-------------|-----------|--------------------------|
| 64    | 128KB       | 0.91      | 7.3                      |
| 128   | 256KB       | 1.4       | 20                       |
| 256   | 256KB       | 1.7       | 46                       |
| 512   | 256KB       | 2.3       | 93                       |

### 4. CGRA (粗粒度可重构阵列)

**代表项目**：ADRES、Samsung SGPU、RIKEN CGRA

**工作原理**：
- 由多个功能单元（FU）组成的互连阵列，每个 FU 可执行 ALU/MAC/Load/Store
- 通过"配置上下文"（Context）在每个周期或每几个周期重新配置 FU 的连接和操作
- 将计算的数据流图（DFG）映射到空间阵列上
- 比 FPGA 粒度大（word-level vs bit-level），配置开销小
- 比 ASIC/脉动阵列灵活，但面积效率稍低

**与 CNN NPU 的关系**：
- CGRA 天然适合不规则计算图（如 Transformer 中的动态路由）
- 对纯 CNN 推理来说"杀鸡用牛刀"——每个 FU 的控制/配置存储占面积
- 映射工具（编译器）复杂度极高，是 CGRA 的核心难点

---

### 5. 混合架构 (Hybrid)

**代表项目**：Google TPU v4（MXU + VPU）、Groq TSP

**常见组合**：
- **脉动阵列 + 向量单元**：脉动阵列处理大矩阵乘法，向量单元处理逐元素运算（ReLU/BN/Pool）
- **数据流 + 可配置后处理**：NVDLA 本身就是混合——卷积核（脉动/数据流）+ SDP/PDP 流水线
- **小脉动阵列 + 可编程 SIMD**：小脉动阵列做 Conv，SIMD 做 Depthwise Conv 和后处理

**小型 NPU 中的混合方案**：
- 最常见：MAC Array（Conv 核心）+ 专用后处理流水线（Activation + Pool + Norm）
- NVDLA 的 "Conv Core + SDP + PDP" 就是典型的混合架构
- Ethos-U55 内部也是 MAC Array + 配套的数据处理流水线

## 三、各架构详细分析

### 3.1 脉动阵列 —— 面积效率之王

**优势**：
1. **极简控制逻辑**：所有 PE 执行相同操作，共享控制信号，控制开销仅占 <1%
2. **规则布局布线**：对 FPGA 和 ASIC 后端都非常友好，时序容易收敛
3. **高计算密度**：PE 面积几乎全是 MAC + 寄存器，利用率最高
4. **可预测性能**：延迟确定性强，适合实时推理场景
5. **成熟开源实现**：Gemmini/tinyTPU 提供完整参考

**劣势**：
1. **Depthwise Conv 灾难**：DW Conv 每通道独立卷积，N×N 阵列中仅 N 个 PE 能同时工作，利用率降至 ~1/N
   - 16×16 阵列做 3×3 DW Conv：利用率仅 9/256 ≈ 3.5%
   - 需要特殊处理（多通道并行、BiRD 数据流等）
2. **启动延迟**：M+N-2 个周期的填充/排空开销，小矩阵时显著
3. **维度不匹配浪费**：当层维度不是阵列尺寸整数倍时，边缘 PE 空闲
4. **单一数据流**：设计时锁定 WS/OS/RS，运行时无法切换

**FPGA 资源估算（0.2 TOPS 目标）**：
```
方案A: 32×32 SA @ 100MHz = 32×32×2×100M = 204.8 GOPS ✓
  - DSP48: 32×32 = 1024 个（INT8 可打包，实际约 512 DSP48）
  - LUT: ~30K-50K（控制逻辑 + 数据路径）
  - BRAM: 64-128 个（输入/权重缓冲）
  - 总计适合 Artix-7 200T 或 Zynq 7045 级别

方案B: 16×16 SA @ 200MHz = 16×16×2×200M = 102.4 GOPS
  需要 2 个阵列分时复用 → 约 512 DSP48
```

**Depthwise Conv 解决方案**：
- **RiSA** (2021)：在脉动阵列上增加旁路数据路径，DW Conv 利用率提升至 60-70%
- **BiRD** (2024)：双向输入复用，DW Conv 利用率达 80%+
- **简单方案**：为 DW Conv 单独设计一个小型并行模块（如 16 通道并行），面积开销约 5-10%

---

### 3.2 SIMD/向量处理器 —— 灵活性之王

**优势**：
1. **可编程性**：通过指令/命令流控制，添加新算子不需要改硬件
2. **DW Conv 友好**：向量单元天然支持逐通道操作
3. **开发门槛较低**：类似 DSP/MCU 的编程模型，软件栈成熟
4. **面积可控**：Ethos-U55 仅 0.1mm²（16nm），适合 MCU 级别

**劣势**：
1. **取指/译码开销**：每条指令都需要取指、译码，增加面积和功耗
2. **MAC 密度稍低**：控制逻辑占比高于脉动阵列
3. **数据复用不如数据流架构显式**：依赖编译器优化
4. **峰值利用率受限**：指令流水线气泡、分支惩罚

**Ethos-U55 MAC 利用率参考**：

| 层类型 | 推荐配置 | 典型利用率 |
|--------|---------|-----------|
| Standard Conv | 128-256 MAC | 60-80% |
| Depthwise Conv | 64-128 MAC | 50-70% |
| Pointwise Conv (1×1) | 32-64 MAC | 40-60% |
| Fully Connected | 128+ MAC | 70-85% |

**对我们项目的问题**：
- Ethos-U55 是闭源 IP，我们无法参考其 RTL
- 纯 SIMD 架构要达 0.2 TOPS 需要非常宽的向量单元（500+ lanes）
- 在 FPGA 上实现完整指令集 + 编译器工作量巨大

---

### 3.3 数据流架构 —— 能效之王

**优势**：
1. **最优数据复用**：通过编译器/映射工具显式规划数据移动路径
2. **能效最高**：Eyeriss 在 65nm 达到 ~0.24 TOPS/W，远超同代方案
3. **灵活的层处理**：不同层可采用不同映射策略
4. **DW Conv 支持较好**：Row Stationary 对 DW Conv 友好

**劣势**：
1. **设计复杂度高**：每个 PE 需要本地存储 + 灵活互连 + 独立控制
2. **NoC 面积开销**：PE 间通信网络占面积 20-30%
3. **映射工具开发难度大**：需要为每种层类型找最优映射
4. **FPGA 上效率损失**：NoC 的灵活互连在 FPGA 上消耗大量路由资源

**NVDLA 作为数据流架构的实际 FPGA 数据**：
```
NVDLA nv_small (64 MAC) @ Zynq 7045:
  - LUT: ~80,000
  - 频率: 100 MHz
  - INT8 性能: 64×2×100M = 12.8 GOPS（远低于我们 200 GOPS 目标）
  - 问题：nv_small 对我们来说太小，而 nv_large/full 塞不进普通 FPGA

如果我们需要 200 GOPS 的 NVDLA:
  - 需要约 1000 MAC @ 100MHz
  - 资源需求：~200K-300K LUT + 大量 BRAM
  - 需要 Zynq UltraScale+ 或更大 FPGA
```

**Eyeriss 规模参考**：
- 168 PE @ 200MHz = 168×2×200M = 67.2 GOPS（65nm ASIC）
- 要达到 200 GOPS 需要约 500 PE @ 200MHz
- 但每个 PE 需要 448B Scratchpad，500 PE → 224KB on-chip SRAM
- PE 间互连网络面积显著

**ShiDianNao 特点**：
- 将 NPU 放在图像传感器旁边，消除 DRAM 访问
- 256 个 16-bit 乘法器，64KB 片上 SRAM
- 65nm 下 4.86 mm²，功耗 320mW
- 能效：~0.6 TOPS/W（16-bit）

---

### 3.4 CGRA —— 灵活性极致但过度设计

**优势**：
1. **极度灵活**：可映射任意计算图
2. **运行时可重构**：不同层可在不同配置间切换
3. **理论上通用**：CNN、RNN、Transformer 都能跑

**劣势**：
1. **面积开销大**：每个 FU 需要配置存储、互连交换、可选操作等，面积是同等 MAC 数脉动阵列的 3-5 倍
2. **映射/编译器极其复杂**：NP-hard 的映射问题，需要启发式求解器
3. **对 CNN-only 严重过设计**：CNN 计算模式规则且可预测，不需要 CGRA 的灵活性
4. **开源工具链不成熟**：缺乏可用的端到端编译器

**对 0.2 TOPS CNN-only NPU 的结论**：❌ 不推荐
- CNN 计算模式高度规则，脉动阵列或简单数据流即可满足
- CGRA 的灵活性溢价在纯 CNN 场景完全浪费
- 编译器开发难度对个人/小团队不现实

## 五、各层类型 MAC 利用率深度分析

### 5.1 Standard Convolution (3×3, stride=1)

**计算特点**：大矩阵乘法（im2col 后），高数据复用

| 架构 | 利用率 | 原因 |
|------|--------|------|
| 脉动阵列 | **75-90%** | im2col 后矩阵尺寸通常匹配良好 |
| SIMD | **65-80%** | 依赖向量化效率 |
| 数据流 (RS) | **80-95%** | 显式优化数据复用 |
| CGRA | **55-70%** | 映射效率受限 |

### 5.2 Depthwise Convolution (3×3)

**计算特点**：每通道独立卷积，计算量小但通道多，无跨通道数据复用

| 架构 | 利用率 | 原因 |
|------|--------|------|
| 脉动阵列 | ⚠️ **3-30%** | 致命弱点！N×N 阵列只能利用 min(K,N) 行 |
| 脉动阵列 + DW优化 | **40-70%** | 需要额外硬件支持（如 BiRD/RiSA） |
| SIMD | **60-80%** | 向量化沿通道维度并行 |
| 数据流 | **50-75%** | Row Stationary 较好支持 |
| CGRA | **50-70%** | 可灵活映射 |

### 5.3 Pointwise Convolution (1×1)

**计算特点**：等价于矩阵乘法 [H×W, Cin] × [Cin, Cout]

| 架构 | 利用率 | 原因 |
|------|--------|------|
| 脉动阵列 | **80-95%** | 纯矩阵乘法，最佳场景 |
| SIMD | **70-85%** | 向量化效率高 |
| 数据流 | **75-90%** | 类似标准 Conv |
| CGRA | **55-70%** | 映射受限 |

### 5.4 Fully Connected

**计算特点**：[Batch, In] × [In, Out]，纯矩阵乘法

| 架构 | 利用率 | 原因 |
|------|--------|------|
| 脉动阵列 | **85-95%** | 完美匹配场景 |
| SIMD | **75-85%** | 向量化良好 |
| 数据流 | **70-85%** | 较大矩阵时好 |
| CGRA | **60-70%** | 映射复杂 |

### 5.5 MobileNet 等轻量模型综合利用率

MobileNet V1/V2 大量使用 Depthwise + Pointwise 组合：
- **脉动阵列（无DW优化）**：综合利用率 ~40-55%（DW 层拖后腿）
- **脉动阵列（有DW优化模块）**：综合利用率 ~65-80%
- **SIMD**：综合利用率 ~60-75%
- **数据流**：综合利用率 ~65-80%

## 六、设计复杂度 & 编译器复杂度评估

### 6.1 设计复杂度（RTL 实现难度）

| 架构 | 估算人月 | 核心难点 |
|------|----------|---------|
| **脉动阵列** | 2-4 人月 | PE 简单规则，难点在数据搬运和 tiling 控制 |
| **脉动阵列 + DW模块** | 3-6 人月 | 增加 DW 专用通路 |
| **SIMD** | 6-12 人月 | 需要设计指令集、流水线、取指/译码逻辑 |
| **数据流 (NVDLA-like)** | 8-16 人月 | 多模块协同、NoC/互连、状态机复杂 |
| **CGRA** | 12-24 人月 | 互连网络 + 配置控制器 + 映射工具 |

### 6.2 编译器/软件栈复杂度

| 架构 | 编译器难度 | 核心工作 |
|------|-----------|---------|
| **脉动阵列** | ⭐⭐ 低 | im2col + tiling + 简单调度，可手写 |
| **SIMD** | ⭐⭐⭐ 中 | 需要完整编译器：算子库 + 指令选择 + 寄存器分配 |
| **数据流** | ⭐⭐⭐ 中 | 层级映射 + 数据搬运调度 + 层融合优化 |
| **CGRA** | ⭐⭐⭐⭐⭐ 极高 | NP-hard 映射问题 + 路由 + 时序约束 |

## 八、推荐方案

### 🎯 推荐：脉动阵列 + Depthwise Conv 专用模块（混合方案）

**理由**：

1. **最佳面积效率**：脉动阵列 PE 面积利用率 > 90%，对 0.2 TOPS 级别刚好在 FPGA 可承受范围
2. **设计复杂度可控**：2-4 人月完成核心阵列，再加 1-2 人月做 DW 模块
3. **编译器简单**：im2col + tiling 即可，无需复杂映射
4. **开源参考丰富**：Gemmini、tinyTPU 提供完整 RTL 和工具链参考
5. **FPGA 友好**：规则结构，时序容易收敛，适合先 FPGA 验证再流片
6. **可扩展**：后续加 MAC 或提频即可线性提升算力

### 推荐配置

```
核心计算架构:
  ┌─────────────────────────────────────────────┐
  │                                             │
  │  ┌───────────────────┐   ┌──────────────┐  │
  │  │  Systolic Array   │   │  DW Conv     │  │
  │  │  16×16 WS         │   │  Module      │  │
  │  │  INT8 MAC         │   │  16-ch       │  │
  │  │  (256 PEs)        │   │  parallel    │  │
  │  └───────────────────┘   └──────────────┘  │
  │                                             │
  │  ┌───────────────────────────────────────┐  │
  │  │  Post-Processing Pipeline             │  │
  │  │  ReLU / Pool / BN / Quantize          │  │
  │  └───────────────────────────────────────┘  │
  │                                             │
  │  ┌───────────────────────────────────────┐  │
  │  │  DMA + Buffer Management              │  │
  │  │  Weight Buffer | Activation Buffer    │  │
  │  └───────────────────────────────────────┘  │
  │                                             │
  └─────────────────────────────────────────────┘

规格:
  - 主阵列: 16×16 Weight Stationary, INT8
  - 频率: 200MHz (FPGA) / 500MHz (ASIC)
  - FPGA 算力: 16×16×2×200M = 102.4 GOPS (单阵列)
  - 双阵列或提频可达 200 GOPS
  - DW 模块: 16 通道并行，3×3 kernel
  - 后处理: ReLU + MaxPool + AvgPool + 量化
  - Buffer: 64KB weight + 32KB activation
```

### 替代方案评估

| 方案 | 适合度 | 理由 |
|------|--------|------|
| **脉动阵列 + DW 模块** | ⭐⭐⭐⭐⭐ 最推荐 | 面积最优，设计简单，开源参考多 |
| **NVDLA small 定制** | ⭐⭐⭐⭐ 备选 | 模块化好，但 RTL 复杂度高，FPGA 资源占用大 |
| **纯 SIMD** | ⭐⭐⭐ 可以 | 灵活但面积效率低，编译器工作量大 |
| **Eyeriss-like 数据流** | ⭐⭐⭐ 可以 | 能效好但设计复杂，NoC 在 FPGA 上低效 |
| **CGRA** | ⭐ 不推荐 | 严重过设计，开发周期太长 |

## 十、总结

对于我们的 **0.2 TOPS、INT8/INT16、CNN-only、MCU 级 NPU**：

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 核心架构 | **脉动阵列** | 面积效率最高，设计最简单，FPGA 友好 |
| 数据流模式 | **Weight Stationary** | 推理最优，控制最简单 |
| DW Conv 方案 | **专用并行模块** | 避免脉动阵列利用率暴跌 |
| 后处理 | **流水线融合** | Conv → ReLU → Pool 一气呵成 |
| 参考设计 | **Gemmini > tinyTPU > NVDLA** | 从成熟度排序 |
| 编译器路线 | **简单 tiling + DMA 调度** | 不需要复杂的通用编译器 |
| 扩展方向 | **增加阵列尺寸 / 提频** | 线性可扩展到 0.5-1 TOPS |
